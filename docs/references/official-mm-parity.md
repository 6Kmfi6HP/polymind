# Official MM Parity Verification

**Status:** Design divergence analysis -- no automated test harness yet
**Date:** 2026-07-05
**Sources:** Polymind `market_making/amm/`, `market_making/bands/` vs `pm-official-mm-keeper` `strategies/amm.py`, `strategies/bands.py`

## Purpose

Document the degree of source parity between Polymind's AMM and Bands strategies
and the official Polymarket MM keeper output (`pm-official-mm-keeper`).

**Important caveat:** The two codebases use fundamentally different mathematical
models and strategy patterns. Parity is not a simple output comparison -- it
requires understanding which design choices diverge and why. This document
catalogs those divergences and identifies what would need to change to converge.

---

## Methodology

For each scenario, we:

1. Define a representative market snapshot (mid price, config parameters)
2. Apply Polymind's pricing/sizing logic with exact formulas
3. Document the expected order output (prices and sizes)
4. Compare against the reference keeper behavior from `pm-official-mm-keeper`

---

## Strategy Architecture Comparison

| Dimension | Polymind | Reference Keeper |
|-----------|----------|------------------|
| **AMM math** | Linear spread ladder with linear-decay sizing | Concentrated liquidity (constant-product x*y=k) with full collateral allocation |
| **AMM price model** | Multiplicative: `price = target * (1 +/- spread%)` | Additive: `price = p_i +/- n*delta` with spread gap, bounded by depth |
| **Bands math** | Multiplicative: `price = target * (1 +/- spread_pct)` | Additive: `price = target - margin` (margin is absolute, not percentage) |
| **Bands sizing** | Single weight per band, full allocation per tick | min/avg/max amount envelope, top-up to avg if below min |
| **Strategy pattern** | Full replace: cancel all, place all | Incremental: cancel excess/out-of-band, place top-ups |
| **Market model** | Single-side market (one price per market) | Binary market (Token A + Token B with complement prices) |
| **Order identity** | Not explicit | `OrderType(price, side, token)` for order-level diffing |
| **Tick rounding** | Yes (configurable tick_size) in AMM; none in Bands | Yes, `round(..., 2)` to 2 decimals in AMM; `MAX_DECIMALS` (6) in Bands |

## Reference Keeper Formulas (for comparison)

### Reference AMM -- Constant Product Concentrated Liquidity

The reference keeper uses a Uniswap-v2-style concentrated liquidity model.

**Sell size** for token at price `p_t`, given inventory `x` and mid price `p_i`:

```
L = x / (1/sqrt(p_i) - 1/sqrt(p_u))
sell_size(p_t) = L / sqrt(p_u) - L / sqrt(p_t) + x
```

**Buy size** for token at price `p_t`, given collateral `y` and mid price `p_i`:

```
L = y / (sqrt(p_i) - sqrt(p_l))
buy_size(p_t) = L * (1/sqrt(p_t) - 1/sqrt(p_i))
```

Where:
- `p_u = min(p_i + depth, p_max)` -- upper price bound
- `p_l = max(p_i - depth, p_min)` -- lower price bound
- `p_min, p_max` -- absolute price bounds from config
- `depth` -- max distance from mid price
- `spread` -- inner gap from mid where no orders are placed
- `delta` -- price increment between levels

Prices are generated by stepping:
```
buy_prices = [p_i - spread, p_i - spread - delta, p_i - spread - 2*delta, ...]  (while >= p_l)
sell_prices = [p_i + spread, p_i + spread + delta, p_i + spread + 2*delta, ...] (while <= p_u)
```

Sizes use the `diff()` function to convert cumulative size to per-level size:
```
diff(arr) = [arr[0], arr[1]-arr[0], arr[2]-arr[1], ...]
```

Collateral is allocated between Token A and Token B using a phi-based formula:
```
phi = (1 / (sqrt(p_i) - sqrt(p_l))) * (1/sqrt(buy_prices[0]) - 1/sqrt(p_i))
```

### Reference Bands -- Additive Margins With Envelope Sizing

Each band has `min_margin`, `avg_margin`, `max_margin` (absolute spreads from target).
Each band also has `min_amount`, `avg_amount`, `max_amount` (size envelope).

Band prices (for buy side):
```
min_price = target - max_margin
buy_price = target - avg_margin
max_price = target - min_margin
sell_price = 1 - target + avg_margin  (binary complement)
```

Strategy is top-up: if current band amount < `min_amount`, place orders to reach
`avg_amount`. If current band amount > `max_amount`, cancel excess.

Bands must not overlap: `band1.min_margin < band2.max_margin` invariant check.

## Polymind Formulas (with exact calculations)

### Polymind AMM -- Linear Spread Ladder

**Pricing:**

For level `i` (0-indexed, `0 <= i < num_levels`):

```
spread_pct_i = min_spread + (max_spread - min_spread) * (i / (num_levels - 1))
buy_price_i  = round(target * (1 - spread_pct_i), tick_size)
sell_price_i = round(target * (1 + spread_pct_i), tick_size)
```

**Sizing** (linear decay):

```
decay = 1.0 - concentration_pct
weight_i = 1.0 - (i / (num_levels - 1)) * (1.0 - decay)
size_i = (weight_i / sum(weights)) * total_exposure
```

**Strategy output** uses `zip(ladder, sizes * 2)` which means sizes are recycled
across the 2*num_levels entries:

| Ladder Entry | Size |
|-------------|------|
| BUY level 1 | size[0] |
| SELL level 1 | size[1] |
| BUY level 2 | size[2] |
| SELL level 2 | size[3] |
| BUY level 3 | size[4] |
| SELL level 3 | size[0] (recycled) |
| ... | ... |

**Notable design quirk:** Because `sizes * 2` concatenates the list, and the ladder
alternates buy/sell per level, the buy and sell at the same level receive *different*
sizes (unless all weights are equal at concentration_pct=0). This means the ladder
is not size-symmetric per level, which may be unintended.

### Polymind Bands -- Multiplicative Spread With Weighted Sizing

**Pricing (per band):**

```
buy_price  = target * (1 - band.spread_pct)
sell_price = target * (1 + band.spread_pct)
```

No tick rounding applied (raw float).

**Sizing (per band):**

```
size = exposure_per_band * band.weight
```

No distribution across bands -- each band independently gets its full allocation.
No top-up/envelope logic: all bands receive orders every tick.

## Scenarios

### Scenario 1: AMM Concentrated Liquidity -- Single Market

**Market snapshot:**
- Mid price: $0.50
- Budget: 100 USDC total exposure per side

**Polymind config (defaults):**
- `min_spread=0.01`, `max_spread=0.05`, `num_levels=5`, `tick_size=0.001`
- `total_exposure=100.0`, `concentration_pct=0.5`

**Pricing computation:**

| Level | Spread % | Buy Price | Sell Price | Formula |
|-------|----------|-----------|------------|---------|
| 1 | 1.00% | 0.495 | 0.505 | `0.01 + 0.04*(0/4) = 0.01` |
| 2 | 2.00% | 0.490 | 0.510 | `0.01 + 0.04*(1/4) = 0.02` |
| 3 | 3.00% | 0.485 | 0.515 | `0.01 + 0.04*(2/4) = 0.03` |
| 4 | 4.00% | 0.480 | 0.520 | `0.01 + 0.04*(3/4) = 0.04` |
| 5 | 5.00% | 0.475 | 0.525 | `0.01 + 0.04*(4/4) = 0.05` |

All prices already at tick_size=0.001 granularity, so no rounding change.

**Sizing computation:**

`decay = 1.0 - 0.5 = 0.5`

| Level | Weight Formula | Weight | Size `= (weight / 3.75) * 100` |
|-------|---------------|--------|-------------------------------|
| 1 | `1.0 - (0/4)*0.5` | 1.000 | 26.667 |
| 2 | `1.0 - (1/4)*0.5` | 0.875 | 23.333 |
| 3 | `1.0 - (2/4)*0.5` | 0.750 | 20.000 |
| 4 | `1.0 - (3/4)*0.5` | 0.625 | 16.667 |
| 5 | `1.0 - (4/4)*0.5` | 0.500 | 13.333 |

Sum of weights = 1.000 + 0.875 + 0.750 + 0.625 + 0.500 = **3.750**

**Expected Polymind order output** (10 orders, sizes recycled via `sizes * 2`):

| Side | Price | Size | Note |
|------|-------|------|------|
| BUY | 0.495 | 26.667 | Level 1 buy (size[0]) |
| SELL | 0.505 | 23.333 | Level 1 sell (size[1]) |
| BUY | 0.490 | 20.000 | Level 2 buy (size[2]) |
| SELL | 0.510 | 16.667 | Level 2 sell (size[3]) |
| BUY | 0.485 | 13.333 | Level 3 buy (size[4]) |
| SELL | 0.515 | 26.667 | Level 3 sell (size[0] recycled) |
| BUY | 0.480 | 23.333 | Level 4 buy (size[1] recycled) |
| SELL | 0.520 | 20.000 | Level 4 sell (size[2] recycled) |
| BUY | 0.475 | 16.667 | Level 5 buy (size[3] recycled) |
| SELL | 0.525 | 13.333 | Level 5 sell (size[4] recycled) |

**Reference keeper equivalent behavior:**

The reference AMM would produce a *completely different* order set because:

1. It uses concentrated liquidity math (`L = y / (sqrt(p_i) - sqrt(p_l))`),
   so sizes depend nonlinearly on price.
2. Prices are generated additively (`price = p_i +/- n*delta`) not as a percentage of mid.
3. It requires binary pair config: Token A and Token B with `p_min`, `p_max`, `delta`, `depth`, `spread`, `max_collateral`.
4. Collateral is allocated across both tokens via the phi formula.
5. A sample reference config (`config/amm.json`) would define `p_min=0.01`, `p_max=0.99`,
   `spread=0.02`, `delta=0.01`, `depth=0.10`, `max_collateral=1000.0`.

For the reference keeper with `target=$0.50`, `spread=0.02`, `delta=0.01`, `depth=0.10`:
- `p_l = max(0.50 - 0.10, 0.01) = 0.40`
- `p_u = min(0.50 + 0.10, 0.99) = 0.60`
- Buy prices: `[0.48, 0.47, 0.46, ..., 0.40]` (9 levels)
- Sell prices: `[0.52, 0.53, 0.54, ..., 0.60]` (9 levels)
- Sizes computed from constant-product formula, varying nonlinearly by level.

**Parity gap:** Fundamental. Polymind AMM is a linear-ladder approximation,
not a constant-product AMM. The two designs serve different risk profiles.

### Scenario 2: AMM Concentrated Liquidity -- Wide Market

**Market snapshot:**
- Mid price: $0.20 (skewed binary market -- e.g., YES at 20% probability)
- Budget: 100 USDC total exposure per side

**Polymind config (defaults):**
- `min_spread=0.01`, `max_spread=0.05`, `num_levels=5`, `tick_size=0.001`
- `total_exposure=100.0`, `concentration_pct=0.5`

**Pricing computation:**

| Level | Spread % | Buy Price | Sell Price |
|-------|----------|-----------|------------|
| 1 | 1.00% | 0.198 | 0.202 |
| 2 | 2.00% | 0.196 | 0.204 |
| 3 | 3.00% | 0.194 | 0.206 |
| 4 | 4.00% | 0.192 | 0.208 |
| 5 | 5.00% | 0.190 | 0.210 |

All prices round cleanly to `tick_size=0.001`.

**Sizing:** Same as Scenario 1 (`total_exposure` and `concentration_pct` unchanged).

**Note:** The **sell levels** at `0.210` with a 5% spread from 0.20 reach only 0.210 --
far from the reference keeper's `p_u = min(0.20 + 0.10, 0.99) = 0.30`. The reference
keeper would place sell orders from 0.22 up to 0.30 (9 levels), covering a much wider
range. Polymind's multiplicative spread creates a narrow absolute range at low prices:
`0.20 * 0.05 = 0.01` absolute width per side vs the reference keeper's fixed `0.10`
depth.

**Parity gap:** The multiplicative spread model means absolute order coverage shrinks
at extreme prices, while the reference keeper's additive depth provides consistent
coverage. This is a deliberate design tradeoff but a source of divergence.

### Scenario 3: Bands Strategy -- 3 Default Bands

**Market snapshot:**
- Mid price: $0.50
- Budget: 20 USDC exposure per band per side

**Polymind config (defaults):**
- `bands = [(spread=1.5%, weight=1.0), (spread=3.0%, weight=1.0), (spread=5.0%, weight=1.0)]`
- `exposure_per_band = 20.0`

**Pricing computation (no tick rounding in bands):**

| Band | spread_pct | Buy Price | Sell Price |
|------|-----------|-----------|------------|
| 0 (inner) | 1.5% | 0.50 * (1 - 0.015) = 0.4925 | 0.50 * (1 + 0.015) = 0.5075 |
| 1 (middle) | 3.0% | 0.50 * (1 - 0.03) = 0.485 | 0.50 * (1 + 0.03) = 0.515 |
| 2 (outer) | 5.0% | 0.50 * (1 - 0.05) = 0.475 | 0.50 * (1 + 0.05) = 0.525 |

**Sizing:** All weights=1.0, so `sizes = [20.0, 20.0, 20.0]`.

**Expected Polymind order output** (6 orders):

| Side | Price | Size | Band |
|------|-------|------|------|
| BUY | 0.4925 | 20.0 | 0 |
| SELL | 0.5075 | 20.0 | 0 |
| BUY | 0.485 | 20.0 | 1 |
| SELL | 0.515 | 20.0 | 1 |
| BUY | 0.475 | 20.0 | 2 |
| SELL | 0.525 | 20.0 | 2 |

**Reference keeper equivalent behavior:**

The reference bands keeper uses *additive* margin (not multiplicative):

```
buy_price = target - avg_margin     (e.g., 0.50 - 0.015 = 0.485 for inner band)
sell_price = 1 - target + avg_margin (binary complement, e.g., 0.50 + 0.015 = 0.515)
```

With reference config bands like `{"min_margin": 0.01, "avg_margin": 0.015, "max_margin": 0.02, "min_amount": 10, "avg_amount": 20, "max_amount": 30}`:

- Inner band buy range: 0.48 to 0.49 (max_margin to min_margin)
- Target buy price: 0.485 (avg_margin = 0.015)
- Top-up: buys if amount < 10, targets 20, caps at 30

Key divergence: reference uses complement for sell prices (`1 - target + margin`),
Polymind uses `target * (1 + spread)`. For target=0.50 these happen to be close,
but at target=0.20:
- Reference: `sell_price = 1 - 0.20 + 0.015 = 0.815`
- Polymind: `sell_price = 0.20 * (1 + 0.015) = 0.203`
These are wildly different because the reference treats bands as binary-market
instruments (Token A at price p, Token B at price 1-p), while Polymind treats
them as single-market spreads.

**Parity gap:** The models are structurally incompatible for non-0.5 prices.
The reference's additive margin with binary complement is designed for prediction
markets where YES+NO = 1. Polymind's multiplicative spread is a generic market-making
construct.

## Current Parity Status

| Scenario | Polymind | Reference | Match? | Notes |
|----------|----------|-----------|--------|-------|
| AMM -- Single Market ($0.50) | 10 orders, linear spread, linear-decay sizes | Non-linear CP AMM, additive prices, phi-based collateral allocation | Different model | Polymind is a simplified ladder; keeper uses CP invariant. Not comparable numerically. |
| AMM -- Wide Market ($0.20) | Narrower absolute spread (max 5 ct above) | Wider depth (10 ct above) | Different coverage | Multiplicative spread shrinks at low prices. |
| Bands -- 3 Bands ($0.50) | 6 orders at multiplicative spreads | 2-6 orders at additive margins with binary complement | Similar structure, different math | At p=0.50, additive and multiplicative give similar but not identical values. |
| Bands -- 4 Weighted Bands ($0.50) | 8 orders with weighted sizes | Envelope min/avg/max per band | Different sizing model | Weight vs envelope are different control paradigms. |
| Full-replace vs incremental | Cancel all, place all | Cancel excess, place top-ups | Different pattern | Full-replace is simpler but wastes cancel/place gas. |

Legend: Match = identical numerical output on same inputs (not expected here due to
fundamentally different models). "Similar structure" = comparable concept but different
math. "Different model" = fundamentally different approach.

**Verdict: No numerical parity exists, nor should it.**

The Polymind AMM and Bands strategies are *design alternatives* to the reference
keeper, not ports. The reference keeper informed the architecture (snapshot-to-orders
split, separation of pricing from execution), but the actual math was redesigned.

## Key Design Divergences

### 1. AMM: Constant Product vs Linear Ladder

The reference keeper uses a true concentrated-liquidity AMM (Uniswap v2 constant
product formula). Polymind uses a linear spread ladder with linear decay sizing.

**Implications:**
- Polymind's sizes are deterministic and independent of price level within a ladder.
  Reference keeper sizes grow hyperbolically as price approaches bounds.
- Polymind requires less collateral data (just total_exposure). Reference keeper
  requires token balances, collateral balance, and allocation across tokens.
- Polymind is simpler and more predictable; reference keeper is capital-efficient
  in the concentrated-liquidity sense.

### 2. Bands: Additive vs Multiplicative Spreads

The reference keeper uses additive margins (`target - margin`), which are natural
for prediction markets where price is probability. Polymind uses multiplicative
spreads (`target * (1 +/- pct)`).

**Implications:**
- At extreme prices (e.g., $0.20), additive margins produce symmetric absolute ranges
  while multiplicative spreads produce asymmetric absolute ranges.
- Multiplicative spreads are more natural for traditional asset market making.
- Additive margins integrate naturally with binary complement pricing.

### 3. Strategy Pattern: Full Replace vs Incremental

Polymind cancels all existing orders and places a fresh ladder on every tick.
The reference keeper computes a diff between expected and current orders, cancelling
only excess/out-of-band orders and placing only top-ups.

**Implications:**
- Full replace is simpler and immune to order drift, but generates more on-chain
  traffic (each tick = cancel N + place N transactions).
- Incremental is gas-efficient but requires order tracking and identity mapping.

### 4. Market Model: Single vs Binary

Polymind treats each market independently with a single mid price. The reference
keeper treats markets as binary pairs (Token A + Token B, price A + price B = 1).

**Implications:**
- Polymind can work with any CLOB market, not just binary prediction markets.
- The reference keeper's complement pricing is more capital-efficient for binary
  markets (sell on one side = buy on the other).
- Polymind's approach is more general; reference's is more optimized for the
  Polymarket domain.

## What Would Be Needed for Numerical Parity

If the goal is to make Polymind AMM/Bands produce *identical* outputs to the
reference keeper on the same inputs, the following changes would be required:

### AMM Parity
1. Replace `compute_ladder` with concentrated-liquidity formulas (constant product).
2. Replace `distribute_size` with `diff(cumulative_size)` pattern.
3. Add phi-based collateral allocation across token pairs.
4. Add binary market complement logic.
5. Replace linear interpolated spreads with additive price stepping (`p_i +/- n*delta`).

### Bands Parity
1. Replace multiplicative spreads with additive margins (`target - margin`).
2. Add min/avg/max amount envelope per band (not just weight).
3. Replace full-replace with incremental top-up/cancel-excess model.
4. Add binary complement pricing for sell orders.
5. Add band overlap validation.

### General
1. Add order identity type (`OrderType(price, side, token)`) for order-level diffing.
2. Switch to incremental strategy pattern.

Note: These would be a *rewrite*, not a refactor. The current design was intentionally
simplified. Parity should only be pursued if the `pm-official-mm-keeper` behavior is
required for production.

## Python Verification Script

The following script can be run to verify Polymind's actual output matches the
expected values documented above:

```python
"""Verify AMM and Bands parity scenario outputs match documented expectations."""

from polymind.strategies.market_making.amm.pricing import AMMPricingConfig, compute_ladder
from polymind.strategies.market_making.amm.sizing import distribute_size
from polymind.strategies.market_making.bands.pricing import BandConfig, BandPricingConfig, compute_band_prices
from polymind.strategies.market_making.bands.sizing import BandSizingConfig, distribute_band_sizes


def test_amm_scenario_1():
    """Verify Scenario 1 AMM output."""
    pcfg = AMMPricingConfig(num_levels=5, min_spread=0.01, max_spread=0.05, tick_size=0.001)
    sizes = distribute_size(total_exposure=100.0, num_levels=5, concentration_pct=0.5)
    ladder = compute_ladder(target_price=0.50, config=pcfg)

    # Verify pricing
    expected_prices = [0.495, 0.505, 0.490, 0.510, 0.485, 0.515, 0.480, 0.520, 0.475, 0.525]
    actual = [(side, price) for side, price, _level in ladder]
    assert len(actual) == 10
    for (_, price), expected in zip(actual, expected_prices):
        assert price == expected, f"Price {price} != {expected}"

    # Verify sizing weights
    expected_weights = [1.0, 0.875, 0.75, 0.625, 0.5]
    for i, (w, ew) in enumerate(zip(sizes, expected_weights)):
        ratio = w / sum(sizes) * 100
        expected_ratio = ew / 3.75 * 100
        assert abs(ratio - expected_ratio) < 0.001, f"Level {i} ratio {ratio} != {expected_ratio}"

    print("AMM Scenario 1: PASS")


def test_bands_scenario_3():
    """Verify Scenario 3 Bands output."""
    bands = [
        BandConfig(spread_pct=0.015, weight=1.0),
        BandConfig(spread_pct=0.03, weight=1.0),
        BandConfig(spread_pct=0.05, weight=1.0),
    ]
    pcfg = BandPricingConfig(bands=bands)
    scfg = BandSizingConfig(exposure_per_band=20.0)

    prices = compute_band_prices(target_price=0.50, config=pcfg)
    sizes = distribute_band_sizes(pcfg, scfg)

    assert len(prices) == 6  # 3 bands * 2 sides
    assert sizes == [20.0, 20.0, 20.0]

    expected_prices = [(band.spread_pct, 0.50 * (1 - band.spread_pct), 0.50 * (1 + band.spread_pct))
                       for band in bands]

    for (side, price, _idx), (spread, buy_exp, sell_exp) in zip(prices, expected_prices):
        expected_price = buy_exp if side.value == "BUY" else sell_exp
        assert abs(price - expected_price) < 1e-10, f"Price {price} != {expected_price}"

    print("Bands Scenario 3: PASS")


if __name__ == "__main__":
    test_amm_scenario_1()
    test_bands_scenario_3()
    print("All parity scenario checks: PASS")
```
